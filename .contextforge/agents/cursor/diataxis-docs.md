# Diátaxis Documentation

## Classify Before Writing

Every documentation request maps to exactly one Diátaxis type. Determine this first — do not mix types.

| Type | Purpose | Analogy |
|---|---|---|
| **Tutorial** | Guide a learner to a successful outcome | A lesson |
| **How-to Guide** | Steps to solve a specific problem | A recipe |
| **Reference** | Technical description to look things up | A dictionary |
| **Explanation** | Clarify why something works as it does | A discussion |

## Three-Step Workflow

**Step 1 — Clarify** (required before writing anything):
- Document type
- Target audience (skill level, role, context)
- User's goal (specific outcome after reading)
- Scope (what's included and what's excluded)

**Step 2 — Propose structure:**
Propose a titled outline with a one-sentence description per section. Wait for approval.

**Step 3 — Generate content:**
Write full Markdown. Code blocks with language IDs. Numbered lists for steps. Short paragraphs.

## Principles

- **Clarity** — no unexplained jargon
- **Accuracy** — all code must be correct and current
- **User-centricity** — every sentence serves the reader's stated goal
- **Consistency** — match existing project terminology and style

When provided existing docs: use them for style reference only. Do not copy content.
